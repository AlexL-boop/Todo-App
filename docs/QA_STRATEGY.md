Стратегия и процессы  
1) Стратегия тестирования
Контекст: команда 5 разработчиков + 2 QA. Цель — быстро выпускать фичи и удерживать качество без “ручной героики”.
Организация процесса

Shift-left: QA подключаются на этапе refinement. На каждую фичу: acceptance criteria + риски + тест-идеи + “что автоматизируем”.
Definition of Ready / Done:
DoR: понятные AC, макеты/контракты API, данные для тестов, нефункциональные ожидания (персистентность, дубли, ограничения ввода).
DoD: пройден smoke, нет блокеров/критов, добавлены/обновлены автотесты на P0, обновлены чек-листы.
Риск-ориентированный подход: P0 — “money/данные/ключевые сценарии”, P1 — edge/валидация, P2 — косметика/редкие кейсы.

Что внедрять первым (приоритет)
E2E Smoke (P0) — 5–10 сценариев “жизнь продукта” (login/основной поток, CRUD ключевых сущностей, критичные интеграции).
Почему: даёт быстрый сигнал “продукт жив” на CI и перед релизом.
API-тесты (контракт/интеграция) — проверка статусов/схем/пагинации/валидации.

Почему: дешевле E2E, быстрее, ловит больше дефектов на ранней стадии.
Компонентные/интеграционные UI-тесты (если фронт сложный) — на критичных формах/валидации.

Почему: закрывает “провалы” между unit и E2E, меньше флейка.
Exploratory + чек-листы регресса — сессии 30–60 минут по рискам спринта.

Почему: ловит классы дефектов, которые тесты не покрывают (UX, визуал, неочевидные комбинации).
Распределение работ QA (2 инженера)
QA1: E2E smoke + поддержка фреймворка/CI + тест-дизайн по P0.
QA2: API/контракт + exploratory + регресс-чек-листы + тестовые данные/окружения.
Ротация раз в 1–2 спринта, чтобы не было bus-factor=1.

2) CI/CD: интеграция автотестов
Принцип: частые быстрые проверки на коммит → более широкие на merge → полный набор перед релизом.
На каждый коммит / PR (быстро, ≤ 10–15 минут)
Lint/format + typecheck
Unit tests
API smoke (критичные эндпоинты/контракты) — если есть API
E2E smoke (самые критичные 5–10 тестов) в Chromium (самый быстрый/стабильный)

Артефакты при падении: trace/video/screenshot + лог network/console
На merge в main / nightly (шире, 20–60 минут)
Полный набор API регресса
E2E regression (все P0/P1)
Прогон кросс-браузер:
минимум: Chromium + Firefox

WebKit — nightly или перед релизом (если не критичен ежедневно)
Параллелизация по шардированию тестов
Перед релизом (полный контроль качества)
Полный E2E regression (Chromium/Firefox/WebKit)
Smoke на staging + sanity по критичным интеграциям

Проверка миграций/совместимости (если релиз включает schema changes)
При необходимости: перфоманс-замеры ключевых страниц (Lighthouse/web-vitals) и “synthetic transactions”
Стратегия стабильности
Ретраи: только на CI (например retries=1), чтобы отлавливать флейк, но не маскировать баги.
Трекинг flaky-тестов: отдельный список + SLA на починку (например, “починить за 48 часов”).
Тесты должны быть детерминированны: тест-данные, предсказуемое окружение, контроль времени/локали при необходимости.

3) Метрики качества
Метрики делю на 3 уровня: продукт, релиз, эффективность QA.
Качество продукта

Escape defects: количество дефектов, ушедших в прод (особенно P0/P1).
Crash-free / error rate (если есть мониторинг).
Время отклика ключевых сценариев (p95) + web-vitals (LCP/CLS/INP) для фронта.
Качество релиза / процесса

Change failure rate: доля релизов, потребовавших hotfix/rollback.
Lead time / Cycle time: от “в работу” до “в прод”.
Defect leakage по этапам: где нашли (dev/QA/staging/prod).
Эффективность QA

Автопокрытие по рискам: P0 сценарии покрыты на 100% (не “по строкам кода”, а по critical flows).
Flaky rate: доля падений “не по вине продукта”.
MTTR тестов/пайплайна: среднее время починки красного CI.
Время обратной связи: сколько минут до первого сигнала (smoke) после PR.
Важно: метрики не должны превращаться в KPI “наказания”. Они нужны, чтобы управлять рисками и скоростью.

Ключевые навыки QA (2024–2025)
Автоматизация на уровне продукта: Playwright/Cypress/Selenium (в идеале Playwright) + умение писать поддерживаемый код (POM, fixtures, стабильные локаторы).

API/контракт тестирование: понимание HTTP, схем, валидации, моков, тестовых данных.

CI/CD мышление: как сделать быстрый, устойчивый пайплайн и диагностировать падения.

Надёжность тестов: устранение флейка, правильные ожидания, наблюдаемость (trace, logs).

Тест-дизайн и риск-ориентированный подход: эквивалентные классы, pairwise, приоритизация.

Коммуникация и продуктовый взгляд: перевод дефектов в риск для бизнеса/пользователя, ясные баг-репорты.

Базовая безопасность и приватность: особенно если есть авторизация/персональные данные.
